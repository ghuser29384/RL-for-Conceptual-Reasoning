# Production Architecture

This project now has a deployable split between the static annotation UI and request-scoped API handlers.

## Runtime Shape

- Static UI: built by `npm run build` into `dist/`.
- Local development server: `node src/server.mjs`.
- Vercel API entrypoints: files under `api/`, all delegating to `src/vercel/api-handler.mjs`.
- Shared API policy: `src/server.mjs`.
- Local audit storage: `src/storage/local-audit-store.mjs`, backed by append-only JSONL files.
- Production audit storage: `src/storage/postgres-audit-store.mjs`, backed by the append-only `audit_events` table in `db/production-schema.sql`.

## Environment

Use these environment variables for production:

- `LMCA_SESSION_SECRET`: required; use a strong random value.
- `LMCA_AUTH_MODE=external_jwt`: enables real identity tokens instead of local demo sessions.
- `LMCA_AUTH_ISSUER`: exact issuer URL from the identity provider.
- `LMCA_AUTH_AUDIENCE`: expected API audience for annotation-platform access tokens.
- `LMCA_AUTH_JWKS_URL`: identity-provider JWKS endpoint for RS256 token verification.
- `LMCA_AUTH_ROLE_CLAIM`: optional, defaults to `lmca_role`. For default simple claim names, the server first reads the top-level JWT claim, then Clerk-style `public_metadata`, `metadata`, or `user_metadata`.
- `LMCA_AUTH_ASSIGNMENTS_CLAIM`: optional, defaults to `lmca_assignments`. For default simple claim names, the same metadata fallback applies.
- `LMCA_REQUIRE_REAL_AUTH=true`: fail closed if the deployment is not configured for external JWT auth.
- `LMCA_AUDIT_STORE=postgres`: routes all persisted rating/audit/certification/benchmark events to Postgres.
- `POSTGRES_URL` or `DATABASE_URL`: Postgres connection string.
- `POSTGRES_CONNECTION_LIMIT`: optional, defaults to `5`.
- `POSTGRES_RLS_ROLE`: optional trusted database RLS role for the server audit adapter, defaults to `service`.
- `POSTGRES_SSL=disable`: optional local-only override.
- `LMCA_REQUIRE_PRODUCTION_STORAGE=true`: fail closed if the deployment is not configured for Postgres.

If `LMCA_AUDIT_STORE` is not `postgres`, Vercel functions fall back to `/tmp/rlhf-audit` with `auditMode=local_jsonl_ephemeral_vercel_preview`. That mode is acceptable for preview demos only; it is not durable production storage.

Local development still defaults to signed demo sessions. In production, set `LMCA_AUTH_MODE=external_jwt` and `LMCA_REQUIRE_REAL_AUTH=true` after the identity provider issues access tokens containing an allowed role and assignment list.

## Database Setup

Apply `db/production-schema.sql` before enabling `LMCA_AUDIT_STORE=postgres`.

The first production-critical table is `audit_events`. It stores append-only event streams for:

- `rating`
- `certification`
- `benchmark_exposure`
- `source_style_audit`
- `workflow`

The remaining tables in the schema are the initial normalized backbone for users, position/critique text versions, assignments, context snapshots, label snapshots, release reports, protected rating projections, workflow-state transition logs, verification/adjudication controls, interaction/practice/UX evidence, metric/governance-control evidence, Phase 1 source intake, source preparation, and the operational-control read model. The rating projection tables are `ratings` and `rating_score_values`. They project validated blind-initial and revision audit events into immutable protected rows, preserving score-input policy ids, raw/displayed score JSON, per-dimension score values, confidence, triggered-explanation provenance, context/text version ids, policy-decision bindings, and lock timestamps without exposing peer/source/protected data to rater-facing SQL paths.

The rating-experience projections `rater_instruction_compatibility_policies`, `rater_instruction_render_versions`, `rubric_copy_traceability_maps`, and `rater_instruction_comprehension_audits` store the submitted instruction-render evidence chain. Compatibility policies freeze split classes, compatible render classes, shared policy fields, thresholds, and merge/sensitivity rules. Render versions preserve the active rubric/workflow/profile bindings, score-input policy, UX policy, pre-submit assist policy, rubric lint config, visual variant, compatibility class, and `unset_required` score-default contract. Rubric-copy traceability maps preserve the covered screens, copy entries, mapped Appendix-F clauses, semantic-drift tests, release-config manifest binding, and reviewer timestamps. The comprehension audit indexes the active render version, compatibility policy, rubric-copy traceability map, covered screens, comprehension methods/checks, glossary terms, disclosure depths, reviewer, and frozen timestamp while enforcing that the artifact passed clause-traceability, screen-copy mapping, comprehension, semantic-drift, feature-loss, and protected-leakage checks. It also enforces `excluded_from_score_computation = true` and `excluded_from_independent_blind_denominator = true`, so this read model supports audit/release evidence without becoming a label, score, rater-count, or live queue table.

Admins and auditors can read those rating-experience resources back as collections through `/api/v1/rater-instruction-compatibility-policies`, `/api/v1/rater-instruction-render-versions`, `/api/v1/rubric-copy-traceability-maps`, and `/api/v1/rater-instruction-comprehension-audits`; the workflow console exposes the same collection selectors so operators can verify submitted instruction-render evidence without relying on ad hoc artifact ids.

The interaction/practice/UX projection tables `session_pacing_policies`, `rater_sessions`, `public_example_practice_sessions`, `practice_sandbox_policies`, `rater_dashboard_policies`, `rater_learning_plans`, `source_anchor_examples`, `ux_simplification_policies`, `ux_simplification_reviews`, `item_issue_quarantine_policies`, `item_issue_reports`, `item_issue_actions`, `screen_feature_parity_checks`, and `simplified_copy_previews` store the submitted pacing, practice, dashboard, source-anchor, simplification, item-issue, parity, and copy-preview evidence chain. Private rater session, practice, dashboard, and learning-plan rows stay admin/auditor SQL readback, while `item_issue_reports` and `item_issue_actions` also allow expert reviewer readback for issue triage. These rows are workflow evidence and collection readback only; they do not create labels, calculate scores, run live queues, or make release-gate decisions without becoming label, score, live queue, or release-gate tables.

The metric/governance-control projection tables `metric_configs`, `derived_utility_formulas`, `pairwise_comparison_snapshots`, `governance_approval_records`, and `protected_artifact_revalidations` store submitted metric-directionality, derived-utility, pairwise-denominator, governance-approval, and protected-artifact revalidation evidence. They preserve policy-decision ids, policy action kinds, affected artifact ids, target label snapshot ids, metric/formula ids, release-config manifest ids, and protected artifact ids for admin/auditor SQL readback. They are not a scoring engine, approval authority, deprotection mechanism, protected-content restore path, or live release gate; the append-only workflow event, policy-decision consumption, and release report remain the authoritative evidence surfaces.

The `workflow_state_transition_logs` table projects accepted and rejected backend-guarded transitions for assignment, rating, discussion, adjudication, verification, snapshot, evaluation, export, and release lifecycles. It stores prior/requested/accepted states, evaluated guard checks, failed guard reasons, lock/freeze artifacts, source/tag/protected-visibility state, actor role, and accepted-or-rejected transition status, so the release-critical state-machine evidence has a production read model instead of relying only on raw workflow events.

The work-unit projections are `positions`, `position_text_versions`, `critiques`, `critique_text_versions`, `assignments`, and `rating_context_snapshots`. They project existing admin workflow events for position intake, critique intake, immutable item-text versions, assignment creation, and rating-context snapshots into the normalized SQL backbone used by RLS and downstream protected reads. Assignment rows preserve the external rater id separately from the optional internal `users.id` UUID, so Clerk/JWT assignment claims can authorize assigned work without requiring every workflow rater id to be a database UUID.

The release-artifact projections include immutable `label_snapshots` and `release_reports` plus operator-evidence projections for submitted release and model-evaluation packages. Submitted label snapshots project into immutable target-label rows keyed by snapshot id, including the target label version, input hash, full snapshot JSON, and workflow event metadata. Public dataset-card and methodology-report submissions project into `public_dataset_documents`, preserving their linked release-object ids, content hash, status, submitted document JSON, and workflow event metadata for admin/auditor SQL readback. Hash-only Dataset v0.1 package-review submissions project into `public_dataset_package_reviews`, preserving package-manifest hashes, review status, manifest metadata, and workflow event metadata without storing raw package file contents or exposing a publish action. Admins can also materialize the current server-derived release report through `/api/v1/release-reports`, which records a content-addressed `releaseReport` workflow event and projects the exact report JSON into `release_reports`. The public-style `GET /api/release/report` endpoint remains read-only and side-effect-free.

`GET /api/v1/release-report-sections` exposes bounded admin/auditor readback rows for evidence ids that appear in the current release report, October checklist, or operator review pointers. It can filter by evidence id, checklist row, exact section status, semantic `status=open`/`status=closed`, section key, artifact type, artifact id, or generic route, and `/api/v1/release-report-sections/{evidenceId}` returns the matching bounded row. Open/closed status is derived from linked open October checklist rows, review pointers/summaries/reasons, and linked operator actions; it does not reinterpret a complete checklist row as open merely because the underlying source-intake or source-preparation section is informationally `not_started` while source-derived work is not applicable. Route filtering matches the section by-id readback route plus the linked operator-action, review-pointer, review-summary, October-checklist, target-gap readback, target-data import, dry-run, validate-only, collection-plan, and template routes, so a copied remediation URL can be traced back to the exact release-report evidence section it comes from. Each row exposes the same bounded `routes`/`routeCount` projection used by filtering and counts, avoiding separate reconstruction from scattered route fields. Operator review pointers whose only broad readback was `/api/release/report` now carry item-level release-section routes so reviewers can inspect one blocker without downloading the full report. These rows are projections only: they do not submit artifacts, collect target data, waive release gates, or create completion claims.

`GET /api/v1/release-artifacts/template` exposes a read-only admin/auditor package template for the six release artifacts that satisfy the release-artifact submission package: label snapshot, corpus manifest, training export, public export manifest, internal export manifest, and release report snapshot. It composes the existing operator-action payload-template and operator-evidence JSONL-template readbacks, supports artifact-kind, evidence-status, semantic status, and route filters, and `/api/v1/release-artifacts/template/{id}` returns a single package row by template id or artifact kind. The endpoint preserves the existing LabelSnapshot, CorpusManifest, TrainingExport, ExportManifest, ReleaseReport, operator-action, and operator-evidence importer architecture; it does not submit artifacts, materialize release reports, waive gates, create a new release-artifact type, or turn unchanged templates into production evidence.

`GET /api/v1/operator-evidence/package-manifest` exposes a read-only admin/auditor execution manifest for the submit-operator-evidence phase. It composes the existing October completion runbook, `/api/v1/operator-evidence/import-jsonl-template`, and `/api/v1/operator-action-items/payload-template` into one ordered plan, putting setup payload templates before package JSONL rows and leaving single-record-only actions outside the package import route. It also previews submit actions still blocked by target-data gaps. The manifest supports `checklistRowId`, `actionId`, `artifactKind`, `manifestStepKind`/`stepKind`, `templateKind`, `executionStatus`, `sourceEvidenceId`/`evidenceId`, `resourceKey`, and `route` filters, and each generated row exposes normalized `routes`/`routeCount` plus aggregate `byAllRoute` counts so operators can inspect package-manifest item, write, route-template, dry-run, validate-only, payload-template, runbook, action-group, readback, and verification routes without reconstructing them from scattered fields. The manifest and each generated row carry the filtered submit-evidence `sourceRunbookGroupRoute` and `sourceActionGroupRoute`/`operatorActionGroupRoute`, so operators can jump back to the existing grouped readbacks without reconstructing filters. Every generated row carries a `packageManifestItemRoute`, and `/api/v1/operator-evidence/package-manifest/{id}` returns one generated manifest step. This route does not add a write path, run models, submit artifacts, waive policy or phase gates, freeze releases, or close checklist rows; completion remains authoritative only after real submissions change `/api/release/report`.

`GET /api/v1/public-dataset-readiness` exposes the RLHF93 first-public-artifact gate for **Metaphilosophy Critique Ratings Dataset v0.1** as bounded admin/auditor rows derived from `/api/release/report`. The projection reuses existing release objects (`ReleaseVersion`, `CorpusManifest`, `LabelSnapshot`, item-text and rating-context manifests, split/export manifests, and submitted release-artifact evidence) and reports required rows for release-cleared position-critique pairs, seven-dimensional labels, confidence/explanation metadata, item text hashes, rating-context snapshots, label snapshot, split manifest, corpus manifest, dataset card, methodology report, hidden/protected exclusions, release freeze, and the public-first ladder. It can filter by gate kind, exact or semantic open/closed status, source evidence id/status, review reason, target gap, downstream artifact, or route, and `/api/v1/public-dataset-readiness/{id}` returns one readiness row. Each readiness row exposes the same bounded `routes`/`routeCount` projection used by route filters, combining its by-id readback, release report, document templates, package-file templates, validation, review-manifest, package-review, release-freeze, and downstream guard routes when applicable.
